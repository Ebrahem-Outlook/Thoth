## Current Model and Tokenizer Implementations

- `ByteTokenizer` is deterministic UTF-8 byte tokenization.
- `BpeTokenizer` is deterministic byte-level BPE with save/load artifact support.
- `RecurrentLanguageModel` is a legacy array-based trainable RNN bootstrap with full sequence gradient updates.
- `TransformerLanguageModel` is an array-based decoder-only Transformer forward foundation. It includes embeddings, RMSNorm, causal attention, RoPE, SwiGLU, residuals, checkpointing, and CPU tests.
- The current Transformer training path updates only `lmHead` and `outputBias`. It does not update embeddings, attention, FFN, or norm parameters, and it does not use TorchSharp/autograd.
- TorchSharp is absent from the solution.

## Runtime Model Selection

- `Thoth.Runtime` wires `self`, `hybrid`, and `neural` providers.
- Hybrid and neural quality-gate legacy checkpoint loading through `ModelCheckpointQualityGate`.
- Production runtime still loads the legacy RNN checkpoint path by default: `data/models/thoth-bootstrap.bin`.
- Transformer generation exists in CLI, not as the production chat runtime path.
- Unqualified checkpoints are blocked from production roles by quality gates.

## Conversation and History Flow

- `ChatOrchestrator.SendAsync` loads or creates a conversation, saves attachments, runs `IUserUnderstandingService`, stores the user message, then either runs tools or calls direct chat.
- Direct reply history is bounded to the last 20 messages, but no typed active task state is persisted or passed.
- `SelfUnderstandingService` receives only `UnderstandingRequest.Text`, attachment content types, and project.
- No conversation task state table or continuation state exists.

## Request Understanding Flow

- `ProgrammingLanguageDetector` supports TypeScript, JavaScript, C#, Python, Java, Go, Rust, SQL, HTML, CSS, and SCSS.
- `ProgrammingLanguage.Cpp` does not exist.
- `CodeRequestAnalyzer` extracts language, artifact, behavior, inputs, output, validation, and repository-bound hints from only the current text.
- Arabic normalization is limited to direct string matches in scattered code; no reusable concept/operation matcher exists.

## Response Contracts from Core to API to Angular

- Core has `AssistantResponseKind` and `MissingDetails` in `AssistantResponse`.
- API `ChatResponseDto` exposes `assistantContent`, `understanding`, `agentRun`, and diagnostics, but not the assistant response kind or missing details.
- Angular transcript renders `message.content` and the side panel renders run diagnostics. It does not receive typed assistant response kind/details from the persisted assistant message.

## Agent Decision and Tool Boundaries

- Tools implement `IAgentTool` and are mediated by `AgentEngine` and sandbox policy.
- Generic code-generation requests should not require tools, but current task state is not strong enough to preserve multi-turn code requirements without reclassifying the second turn.

## Checkpoint Formats and Quality Gates

- `ModelCheckpoint` stores legacy RNN weights and optimizer state.
- `TransformerCheckpoint` stores Transformer weights and optimizer moments for the currently trainable head/bias.
- Sidecar metadata is stored at `<checkpoint>.metadata.json`.
- Quality gate statuses include missing, loading failed, unqualified, and role-qualified states.
- Generation-health/no-leak scores are currently simple evaluator scores, not measured from real generated benchmark outputs.

## Training Options Actually Used

- `TrainingOptions.GradientAccumulationSteps` is declared and validated but not used by either trainer.
- `TransformerConfig.Dropout` is declared and saved in checkpoint metadata but dropout is not applied in the array Transformer.
- RNN trainer uses sequence length, epochs, steps per epoch, LR schedule, weight decay, gradient clipping, seed, and checkpoint cadence.
- Transformer trainer uses sequence length, batch size, epochs, steps per epoch, LR schedule, weight decay, gradient clipping, seed, and checkpoint cadence, but not full-parameter gradients.

## Evaluation Metrics Actually Measured

- `LanguageModelEvaluator` computes average loss/perplexity over token sequences for RNN and Transformer.
- It returns `generation_health = 1.0` for finite loss and `no_internal_leak = 1.0`.
- It does not currently generate benchmark outputs to measure no-leak, language match, clarification usefulness, code compile rate, or tool-schema success.

## Dataset Paths and Exclusions

- Default data root is `data`.
- Default training root is `data/training`.
- `CorpusLoader` skips build/generated/private-ish directories such as `.git`, `.vs`, `.idea`, `bin`, `obj`, `node_modules`, `dist`, `coverage`, `memory`, `uploads`, `models`, `checkpoints`, `tokenizers`, `artifacts`, and `reports`.
- It skips binary/media/archive extensions such as `.db`, `.sqlite`, `.bin`, `.dll`, `.exe`, `.pdb`, images, PDFs, and archives.
- There is no approved-source acquisition pipeline, license verifier, PII scanner, secrets scanner, near-dedupe, contamination scan, or attribution bundle.

## Explicit Known-Issue Verification

1. `ProgrammingLanguage` lacks C++ aliases: verified in `ProgrammingLanguageDetector`.
2. `SelfUnderstandingService` classifies only current text and does not receive active task state/history: verified in `SelfUnderstandingService.UnderstandAsync`.
3. `DirectReplyModelInput` contains current text but no structured active task: verified in `ModelRequestContracts` and `ChatOrchestrator`.
4. `UsefulResponseFallback` supports only narrow calculator cases in C# and TypeScript: verified in `UsefulResponseFallback`.
5. No persisted conversation task state exists: verified by `IConversationStore` and SQLite schema.
6. No continuation/slot-merging engine exists: no matching types/services found.
7. `AssistantResponseKind` and suggested details are lost before API/Angular response: verified by API DTO and Angular models.
8. The array Transformer trains only `lmHead` and `outputBias`: verified in `TransformerLanguageModel.TrainBatch`.
9. TorchSharp is absent: no package references found.
10. `GradientAccumulationSteps` is declared but not used: verified by `TrainingOptions` and trainers.
11. Transformer dropout is declared but not implemented: verified by `TransformerConfig` and `TransformerLanguageModel`.
12. Production runtime loads only the legacy RNN checkpoint path: verified in runtime/options defaults.
13. Transformer runtime generation is CLI-only or incomplete: verified in CLI and runtime registrations.
14. Generation-health/no-leak model metrics are not based on real generated benchmark outputs: verified in `LanguageModelEvaluator`.
15. No approved-source acquisition pipeline exists: no data acquisition project or scripts found.

## Baseline Conclusion

The product shell and safety boundaries are functioning, but v0.5 work must start by adding typed persistent task state and continuation resolution. The first acceptance target is the Arabic multi-turn C++ calculator flow, with C++ language detection, Arabic operation normalization, deterministic slot merging, and a clean user response that does not expose internal diagnostics.
